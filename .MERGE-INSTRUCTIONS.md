## Merge Process (follow this order every time)

### Step 1 — Create the merge branch immediately

The branch must exist before any discussion, research, or planning begins. This gives git a clean
point to commit the plan doc to, and ensures nothing is accidentally done on `aws-corp` or `main`.

```bash
cd D:/.ai/research/omp/.oh-my-pi
git fetch origin
git log --oneline aws-corp..origin/main | head -20  # how much changed?
git show origin/main:package.json | python3 -c "import json,sys; print(json.load(sys.stdin).get('version','?'))"

# Create branch from aws-corp tip (NOT from upstream)
git checkout aws-corp
git checkout -b merge/vX.Y.Z    # use the upstream version number
git branch --show-current       # confirm
```

The branch is named `merge/vX.Y.Z` where `X.Y.Z` is the upstream version being absorbed.
**The branch is created on `aws-corp`, not on `origin/main`.**

### Step 2 — Research phase (no source changes)

With the branch created, investigate what changed upstream:

```bash
# What files changed and how much?
git diff aws-corp..origin/main --stat | sort -rn -k3 | head -40

# Which of OUR files were also touched upstream?
git diff aws-corp...origin/main --name-only | python3 -c "
import sys, subprocess
for f in sys.stdin.read().splitlines():
    a = subprocess.run(['git','show',f'aws-corp:{f}'], capture_output=True).returncode == 0
    b = subprocess.run(['git','show',f'origin/main:{f}'], capture_output=True).returncode == 0
    if a and b: print('BOTH:', f)
    elif a: print('OURS:', f)
" | head -40

# New features / breaking changes
git log --oneline aws-corp..origin/main | grep -iE "feat|break|remove|delete" | head -20

# Specific file diffs for our high-risk files
git diff aws-corp..origin/main -- packages/ai/src/providers/amazon-bedrock.ts | head -60
git diff aws-corp..origin/main -- packages/coding-agent/src/slash-commands/builtin-registry.ts | head -40
```

Ask the user questions during this phase. Do not start editing source files.

### Step 3 — Write `.merge_vX.Y.Z.md` and commit it

The plan document goes in the repo root and is excluded from git by `.gitignore` (`/.*` pattern).
Force-add it:

```bash
# Write the file (see §Plan Document Contents below for what must be in it)
# Then:
git add -f .merge_vX.Y.Z.md
git commit -m "docs: add merge/vX.Y.Z plan and conflict analysis"
```

**The plan doc must be committed before the merge starts.** Amend the commit as the discussion
continues (`git add -f .merge_vX.Y.Z.md && git commit --amend --no-edit`).

### Step 4 — Discuss and finalize the plan

Go through the plan doc with the user:
- Confirm strategy (merge vs rebase — see §Strategy note in plan)
- Confirm open decisions (credential flow, new features to disable, config changes)
- Get explicit approval to start

**Do NOT start the merge until the user says "proceed" or equivalent.**

### Step 5 — Execute the merge

```bash
git merge origin/main --no-ff -m "feat(aws-corp): merge upstream vX.Y.Z"
# Resolve conflicts per plan doc guidance
# After merge: take upstream lock files wholesale
git checkout origin/main -- bun.lock Cargo.lock
# Run silent auto-merge verification grep block from plan doc
# Build, test, deploy
```

### Step 6 — Merge into aws-corp (user approval required)

```bash
# Only after explicit user approval:
git checkout aws-corp
git merge --ff-only merge/vX.Y.Z
# Do NOT push without a separate explicit instruction
```

## Plan Document Contents (`.merge_vX.Y.Z.md`)

Every `.merge_vX.Y.Z.md` must contain all of the following before the merge starts:

| Section | What goes in it |
|---|---|
| **Strategy** | Merge vs rebase, rationale, `--ours`/`--theirs` direction |
| **File-by-file conflict analysis** | Every file both sides touched: our changes vs upstream changes, hunk line numbers, overlap risk, resolution rule |
| **Silent auto-merge risk** | Files with disjoint hunks — list the grep verification commands |
| **Engineering work** | Files requiring real code changes (not just conflict resolution): what to add, exact code snippets |
| **New upstream features** | What each new feature does, its `.enabled` flag, whether corp needs to disable it |
| **Config file updates** | All 6 config files, exact new keys, corp-specific overrides |
| **Post-merge checklist** | Numbered steps: lock files, compile, tests, verify greedy-silent files, build, deploy |
| **Open decisions** | Anything that needs user input before merge can start — answer these BEFORE step 5 |

The plan is the contract. If something isn't in the plan, it wasn't discussed. If it wasn't discussed, it doesn't happen.

## Commit Rules

- **NEVER commit without explicit user approval in the current turn.**
- Permission granted in one turn does NOT carry to the next. Each commit needs its own instruction.
- The only exception: committing the `.merge_vX.Y.Z.md` plan doc (step 3 above) — this is pre-approved as part of the process.
- **NEVER push without a separate explicit instruction.** "Commit" and "push" are different instructions.
- Do not use `git checkout HEAD -- <file>` during a merge — HEAD may point to upstream tip, destroying work. Use `git checkout aws-corp -- <file>` or `git checkout origin/main -- <file>` explicitly.

## Lessons Learned

### 1. Stash working-tree docs before starting the rebase
The rebase will abort with "cannot rebase: you have unstaged changes" if any tracked
files are modified but not committed. Commit or stash ALL working-tree changes first:
```bash
git stash push -m "pre-rebase-working-tree"
git rebase origin/main
git stash pop   # after rebase completes
```
Docs like .MERGE-INSTRUCTIONS.md and merge_<version>.md should be committed ON the
merge branch before the rebase starts, not left as working-tree files.

### 2. --ours/--theirs are INVERTED during rebase vs merge
During `git rebase origin/main`:
- `git checkout --ours file`   = takes origin/MAIN version (HEAD = upstream)
- `git checkout --theirs file` = takes the aws-corp COMMIT being replayed

This is the OPPOSITE of `git merge`, where `--ours` = your branch.
**Rule: during rebase, to keep aws-corp's version: use `--theirs`. To keep main's version: use `--ours`.**

Automated resolution scripts that regex-parse conflict markers are safer:
- `<<<<<<< HEAD` block = main's version
- `>>>>>>> <SHA>` block = aws-corp's version
Always parse and take the correct block explicitly rather than relying on --ours/--theirs.

### 3. context-manager.ts: pre-merge analysis said LOW, was actually REAL CONFLICT
Both sides modified the same ~65-line `_rewriteImports()` function block.
Lesson: "different sections of the file" claims should be verified with actual line-number
diffs before the merge, not assumed. The deep-dive subagents caught this.
Resolution: take main's Babel AST rewrite; it subsumes aws-corp's deletion intent.

### 4. stream.ts: eager imports vs lazy register-builtins pattern
aws-corp imports provider stream functions eagerly (needed for aws-corp's `serviceProviderMap`
entry). main moved these to a lazy `register-builtins.ts`. Taking aws-corp's imports produced
TS2393 duplicate identifier errors because the lazy re-export also declares them.
Fix: keep only `import type { AwsCorpOptions }` for main's providers; keep only
`import { streamAwsCorp }` for our own. The builtins register lazily; we register eagerly.

### 5. models.ts: "both" is a valid conflict strategy
When two sides added SEQUENTIAL non-overlapping blocks to the same region, taking "both"
(HEAD block then OUR block) is the correct resolution rather than picking one side.
Used for models.ts: main's `getBundledModel` refactor + aws-corp's aws-corp model block.

### 6. settings-schema.ts memory.backend enum auto-merged
Main added `"hindsight"`, aws-corp added `"mmemory"` to the same `values: [...] as const`
array. Git auto-merged correctly because the additions were at different positions.
No manual intervention needed. Worth verifying in the next merge — may not always be true.

### 7. Python test suite catches real regressions across merge
test_mmemory.py found 5 real bugs in mmemory_server.py introduced by the merge context
changes (end_ts field handling, mode variable scope, non-session source parsing).
Always run the Python suites immediately after the rebase, before fixing TS errors.
Python errors are faster to diagnose and fix than TS cascades.

### 8. Delegate TS error fixing to parallel subagents
After rebase, 120 TS errors across ~12 files resolved in two parallel batches (4+3 agents).
Each subagent owns a disjoint file set. Pattern: group by file, assign one agent per cluster,
run all in parallel, final tsc check at the end. Faster than sequential fixing by the master agent.

### 9. Pre-merge diff analysis (merge_<version>.md) is essential
The deep-dive analysis caught: context-manager.ts real conflict, m-utils.ts false positive,
settings-schema.ts enum auto-merge, session-stats safe drop, mreview false attribution.
Without it, we would have taken --ours/--theirs blindly on 9 files and introduced
silent regressions. Write and validate the analysis doc BEFORE starting the rebase.

### 10. GIT_EDITOR=true for non-interactive rebase --continue
`git rebase --continue` opens an editor for the commit message. In automation:
```python
env = {**os.environ, 'GIT_EDITOR': 'true'}
subprocess.run(['git', ..., 'rebase', '--continue'], env=env)
```
`GIT_EDITOR=true` accepts the default message without blocking.

### 11. v15.0.0: SlashCommandSpec handle → handleTui split
v15.0.0 split `SlashCommandSpec` into two handler shapes:
- `handle(command, SlashCommandRuntime)` — ACP/RPC mode, no `.ctx`
- `handleTui(command, TuiSlashCommandRuntime)` — interactive TUI, has `.ctx: InteractiveModeContext`

Our mmemory/mprune/mreview commands use `runtime.ctx.showStatus()`, `runtime.ctx.sessionManager`,
etc. — TUI-only features. They MUST be registered under `handleTui`, not `handle`.
Symptom if wrong: commands silently do nothing in TUI (no error, no output).

### 12. v15.0.0: --no-memory must use settings.override, not settings.set
`Settings.set()` calls `#queueSave()` → writes to disk permanently.
`Settings.override()` writes to `#overrides` only, never saves.
The `--no-memory` flag must be session-scoped; using `set` corrupts
`agent-work/config.yml` with `mmemory.enabled: false` for all future sessions.
Symptom: `mmemory.enabled` flips to `false` in config after any `--no-memory` child.

### 13. mtuicontrol spawn queue stalls
`mcommand` queues slash commands via SCHEDULE_SLASH_CHANNEL, processed between agent turns.
Batching `spawn` + `wait` in the same turn causes stalls: `wait` blocks the queue for
its full timeout when the child is already idle (waitForIdle waits for next turn_end
which never fires). Always confirm each follow-up before scheduling the next command.
The `wait` command only resolves early if the child is actually processing something.

### 14. mtuicontrol session config: use ow, not o
The personal `o` session config is at `~/.omp/agent\config.yml` (NOT `~/.omp/config.yml`).
Both `o` and `ow` configs must have `mtuicontrol.enabled: true` for headed tests to work.
Verify both paths — do not assume global config applies.

### 15. mtuicontrol: pi.registerCommand bypassed by SCHEDULE_SLASH_CHANNEL
The `SCHEDULE_SLASH_CHANNEL` → `editor.onSubmit` path routes exclusively through
`executeBuiltinSlashCommand` (BUILTIN_SLASH_COMMAND_REGISTRY). Extension commands
registered via `pi.registerCommand` are only reachable via `session.sendPrompt` →
`#tryExecuteExtensionCommand` — a path that is never invoked from `editor.onSubmit`.

**Symptom:** `/mtuicontrol spawn` returns nothing; zero log entries for the handler.
`/mreview` (builtin) works fine from the same mcommand path.

**Fix:** Register `mtuicontrol` in `builtin-registry.ts` as a `handleTui` entry
(same as `mreview`, `mmemory`, `mprune`). Use `runtime.ctx.session.followUp(result)`
to deliver the response. Remove `pi.registerCommand` and the empty extension factory
from `corp-sdk-extensions.ts`.

### 16. stop --all was not implemented — only stopped the last session
The `stop` case checked `sessionPool.has(parts[1])` to see if parts[1] was a valid session ID.
For `stop --all`, parts[1] = "--all", which is never in the pool, so `id = undefined`
→ `resolveSession(undefined)` → last session only. The `--all` flag was silently ignored.

**Symptom:** Two sessions in pool, `stop --all` follow-up says "Session rpc-X stopped." (only one);
log shows only one `stopped` entry; other session window stays open.

**Fix:** Add explicit `parts.includes("--all")` guard at the top of the stop case that
iterates `sessionPool` and stops all sessions. Also added `!maybeId.startsWith("-")`
guard to the single-stop id-resolution to prevent future flags from being misinterpreted as IDs.

### Native addon distribution — ship alongside binary, do NOT embed

**Changed from v15.0.0:** The native `.node` file is no longer embedded in the binary.

**Why:** `dlopen` requires a real filesystem path. The embedded blob path (`B:\~BUN\native\`)
is a virtual filesystem — `require()` refuses to dlopen from it. Embedding adds ~110 MB
to the binary and requires an extraction step on first run that creates a stale cache
invalidation problem (`__piNativesVX_Y_Z` sentinel mismatch across deploys).

**New distribution model:**
- `omp.exe` (125 MB) — the TypeScript binary, no `.node` embedded
- `pi_natives.win32-x64-modern.node` (110 MB) — shipped alongside in the same directory

The loader at `packages/natives/native/loader-state.js` already checks `execDir`
(`C:\Users\common\AppData\Local\omp\`) as a candidate path before the virtual blob path.

**`deploy.cmd` now copies both files.** `build.cmd` resets `embedded-addon.js` to the
null stub by default. Use `build.cmd --embed` only for single-file internet distribution.

**After Rust builds, embedded-addon.js MUST be reset before compiling binary:**
```bash
bun --cwd=packages/natives run embed:native -- --reset
```
If this is skipped, the binary will bundle the `.node` again (large binary, extraction on launch).

**Rust build is now default (`BUILD_NATIVE=1`). Skip with `--skip-native` if `.node` files are fresh.**
The Rust compile takes ~5 minutes per variant (baseline + modern). Only rebuild when:
- `crates/pi-natives/` or `crates/pi-ast/` Rust source changed upstream
- Version sentinel changed (always changes on version bump — `__piNativesVX_Y_Z`)

**Version sentinel** is compiled into Rust and exported as `__piNativesVX_Y_Z`.
The loader validates it on load. Stale `.node` = crash on startup.
To check what sentinel a `.node` file has:
```python
data = open('pi_natives.win32-x64-modern.node', 'rb').read()
# sentinel may be anywhere in the 110 MB file — scan the whole thing:
print('15_1_3 present:', b'__piNativesV15_1_3' in data)
```

## aws-corp / Bedrock Signing — Complete Root Cause Record

This section documents every bug found and fixed during the v15.1.3 merge bedrock integration,
so future merges don't repeat the same debugging cycle.

### The invariant (verified via live AWS error messages)

AWS Bedrock's SigV4 verifier applies **one encode pass** to the URL path it received before
computing its canonical request. Our signer must do the same pass. The correct chain is:

1. **HTTP URL path:** `encodeURIComponent(modelId)` → e.g. `arn%3Aaws%3Abedrock%3A.../inference-profile%2F...`
2. **`urlPath` passed to `signRequest`:** same as above (pre-encoded)
3. **`canonicalPath` inside `signRequest`:** calls `encodeRfc3986` on each segment, encoding
   the `%` signs: `%3A` → `%253A`, `%2F` → `%252F`
4. **AWS verifier:** receives the URL with `%3A`, applies one encode → `%253A`
5. Both sides produce **identical** canonical strings → signature matches

This is identical to what `@aws-sdk/client-bedrock-runtime` did internally.

**Never change `canonicalPath` to return the path verbatim.** It must re-encode.
**Never pass the raw (unencoded) modelId to `signRequest`.** Always pass the `encodeURIComponent` version.

### Proof

AWS returns the expected canonical string in the 403 error body when the signature is wrong:
```
{"message": "... The Canonical String for this request should have been\nPOST\n/model/us.anthropic...v1%253A0/converse-stream\n..."}
```
`%253A` = double-encoded colon. Our signer must produce this.

### Bug: `$flag("AWS_BEDROCK_SKIP_AUTH")` overrode injected credentials

`streamBedrock` had:
```typescript
if ($flag("AWS_BEDROCK_SKIP_AUTH")) {
    credentials = { accessKeyId: "dummy-access-key", ... };
} else {
    credentials = options.credentials ?? await resolveAwsCredentials(...);
}
```
When `AWS_BEDROCK_SKIP_AUTH` was set (e.g. in test environments), it overrode the real
credentials injected by `aws-corp.ts` with dummy ones → 403.

**Fix:** Injected credentials always take priority:
```typescript
if (options.credentials) {
    credentials = options.credentials; // injected from aws-corp — always wins
} else if ($flag("AWS_BEDROCK_SKIP_AUTH")) {
    credentials = { accessKeyId: "dummy-access-key", secretAccessKey: "dummy-secret-key" };
} else {
    credentials = await resolveAwsCredentials(...);
}
```

### Bug: `process.env.AWS_BEDROCK_DEBUG` frozen by Bun bundler

The Bun `--define "process.env.X=..."` flags freeze specific `process.env.*` expressions
at compile time. `process.env.AWS_BEDROCK_DEBUG` was always `undefined` in the binary.
Use `$flag("AWS_BEDROCK_DEBUG")` which reads from `Bun.env` at runtime.

### Bug: `streamAwsCorp` called `ensureCredentials()` fire-and-forget

Old code:
```typescript
export const streamAwsCorp = (...) => {
    ensureCredentials(); // not awaited — _corpCreds may still be null
    const creds = _corpCreds; // null on first call
    return streamBedrock(..., { credentials: creds ?? undefined }); // no creds!
};
```
Fixed by awaiting inside an async IIFE that pipes to an `AssistantMessageEventStream`:
```typescript
export const streamAwsCorp = (...): AssistantMessageEventStream => {
    const outer = new AssistantMessageEventStream();
    (async () => {
        const ok = await ensureCredentials();
        const creds = _corpCreds;
        // ... log, validate, then:
        const inner = streamBedrock(..., { credentials: creds });
        for await (const event of inner) outer.push(event);
    })();
    return outer;
};
```

### Debugging approach for future 403s

1. Set `AWS_BEDROCK_DEBUG=1` in the deployed `ow.cmd` and `ow` launchers.
2. Restart `ow --new`. Look for:
   - `[aws-corp] streamAwsCorp: creds=ok accessKeyId=... sessionToken=present` — credentials resolved
   - `[bedrock] url=... accessKeyId=ASIA... sessionToken=present(...) source=injected` — real creds reaching signer
   - If `accessKeyId=dummy-ac` → `AWS_BEDROCK_SKIP_AUTH` is set; injected creds not winning
   - If `sessionToken=MISSING` → credentials resolution failed
3. If 403 persists with correct creds, AWS returns the expected canonical string in the error body.
   Extract it: `JSON.parse(errorBody).message` contains `\nThe Canonical String for this request should have been\n'POST\n/model/...`.
   Compare path in canonical string vs URL path to diagnose encoding mismatch.

## SCHEDULE_SLASH_CHANNEL — Must Be Preserved in interactive-mode.ts

### What it is

`SCHEDULE_SLASH_CHANNEL` is an event-bus constant defined in
`packages/coding-agent/src/utils/event-bus.ts` (our file, never touched by upstream):

```typescript
export const SCHEDULE_SLASH_CHANNEL = "tui:schedule-slash";
```

It is the bridge between tool-layer code and the TUI slash-command dispatcher.
All of the following emit on this channel:

| Emitter | What it schedules |
|---|---|
| `packages/coding-agent/src/tools/mcommand.ts` | Any slash command the LLM requests via the `mcommand` tool |
| `packages/coding-agent/src/tools/mmemory/recall-tool.ts` | `/mmemory recall ...` |
| `packages/coding-agent/src/tools/mmemory/reflect-tool.ts` | `/mmemory reflect ...` |
| `packages/coding-agent/src/tools/mmemory/tool.ts` | `/mmemory ...` |
| `packages/coding-agent/src/tools/mreview/tool.ts` | `/mreview <file>` |

### The listener — MUST be in interactive-mode.ts

Upstream deleted this listener when they removed `scheduleInput()`. It must be restored
after every merge. This is the exact code from aws-corp (verified verbatim match):

```typescript
// In the constructor, inside if (eventBus) { ... }
// Add to the #eventBusUnsubscribers.push(...) call alongside LSP_STARTUP_EVENT_CHANNEL:
eventBus.on(SCHEDULE_SLASH_CHANNEL, data => {
    const command = typeof data === "string" ? data : null;
    logger.debug("[interactive] SCHEDULE_SLASH_CHANNEL", { command, hasOnSubmit: !!this.editor.onSubmit });
    if (command) void this.session.waitForIdle().then(() => {
        logger.debug("[interactive] SCHEDULE_SLASH executing", { command });
        this.editor.onSubmit?.(command);
    });
}),
```

Import: add `SCHEDULE_SLASH_CHANNEL` to the event-bus import line:
```typescript
import { type EventBus, PIPE_TUI_OUTPUT_CHANNEL, SCHEDULE_SLASH_CHANNEL, type TuiOutputPayload } from "../utils/event-bus";
```

### How to verify after every merge

```bash
grep -n "SCHEDULE_SLASH_CHANNEL" packages/coding-agent/src/modes/interactive-mode.ts
# Must return 2 hits: import line + listener registration
# Zero hits = mcommand, mreview, mmemory tools are all silently broken
```

### Symptom when missing

- `mcommand` tool returns `Scheduled: /mreview ...` but nothing happens
- `/mreview` never opens
- `/mmemory recall` via tool never fires
- No `[interactive] SCHEDULE_SLASH_CHANNEL` entries in the session log
- No error — completely silent failure

### Why upstream removed it

Upstream deleted `scheduleInput()` from `InteractiveModeContext` and the
`SCHEDULE_SLASH_CHANNEL` listener together. They had no equivalent feature.
Our tools continue to emit on this channel. The listener must survive every merge.
