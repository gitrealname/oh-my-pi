## The Big Change: aws-corp → Bedrock Authentication Redesign

This is the **primary engineering task** of this merge — the rest is mechanical conflict resolution.

### What upstream did

| File | Change |
|---|---|
| `amazon-bedrock.ts` | Full rewrite: dropped `@aws-sdk/client-bedrock-runtime`, `@smithy/*`, `proxy-agent`. Native SigV4 + native event-stream decoder. `BedrockOptions` lost `modelIdOverride`. |
| `aws-credentials.ts` | **NEW** — full credential chain: env vars → `~/.aws/credentials` (static/SSO/`credential_process`) → EC2 IMDSv2. Cached per-profile with 60 s skew buffer. |
| `aws-sigv4.ts` | **NEW** — pure SigV4 signing, no SDK dependency. |
| `aws-eventstream.ts` | **NEW** — native `application/vnd.amazon.eventstream` decoder. |
| `aws-corp.ts` | **DELETED** by upstream. |
| `stream.ts` | Removed `streamAwsCorp` import + `aws-corp` routing + `serviceProviderMap` entry. |

### What we need to preserve

1. **Corp OIDC device-auth login flow** — upstream `readSsoCredentials` reads cached tokens from `~/.aws/sso/cache/*.json` but throws if expired ("Run 'aws sso login' first"). Our corp flow *initiates* the device-auth challenge in-process. This must be preserved.

2. **Inference profile ARN construction** — `toInferenceProfileArn()` builds `arn:aws:bedrock:{region}:{accountId}:inference-profile/{modelId}`. Required for corp. Upstream removed `modelIdOverride` from `BedrockOptions`.

3. **`aws-corp` routing in `stream.ts`** — `model.provider === "aws-corp"` must still route to our flow.

### Plan: keep `aws-corp.ts`, adapt it to the new bedrock stack

The cleanest approach that requires the smallest delta and zero changes to upstream files:

#### A. Add `credentials?` injection to `BedrockOptions` (in `amazon-bedrock.ts`)

```typescript
export interface BedrockOptions extends StreamOptions {
    region?: string;
    profile?: string;
    /** Pre-resolved credentials. When set, skips resolveAwsCredentials entirely. */
    credentials?: { accessKeyId: string; secretAccessKey: string; sessionToken?: string };
    /** Override the model ID / inference profile ARN sent to Bedrock. */
    modelIdOverride?: string;
    toolChoice?: "auto" | "any" | "none" | { type: "tool"; name: string };
    reasoning?: Effort;
    // ...
}
```

Then in `streamBedrock`, at the credential resolution point (currently `resolveAwsCredentials({ profile, region, signal })`):

```typescript
const credentials = options.credentials
    ?? await resolveAwsCredentials({ profile: options.profile, region, signal: options.signal });
```

This is a **6-line additive change** to upstream's new `amazon-bedrock.ts` — no conflict surface. For `modelIdOverride`, the URL construction in `streamBedrock` also needs updating (upstream hardcodes `model.id`):

```typescript
// replace the single model.id line in URL construction and request body with:
const effectiveModelId = options.modelIdOverride ?? model.id;
const url      = `https://${host}/model/${encodeURIComponent(effectiveModelId)}/converse-stream`;
const urlPath  = `/model/${encodeURIComponent(effectiveModelId)}/converse-stream`;
// in the request body:
    model: effectiveModelId,
```

Total delta to `amazon-bedrock.ts`: **6 lines** added on top of taking the file wholesale from upstream.

#### B. Keep `aws-corp.ts` OIDC flow as-is, adapt calling convention only

`aws-corp.ts` keeps all `@aws-sdk/credential-provider-sso` and `@smithy/shared-ini-file-loader` imports — **these are not the packages upstream removed**. Upstream only removed `@aws-sdk/client-bedrock-runtime` and `@smithy/client-bedrock-runtime`. The credential provider is a separate package and stays.

The only changes to `aws-corp.ts`:

1. Keep the OIDC device-auth flow verbatim — no changes to the login logic.
2. Keep `toInferenceProfileArn()` verbatim.
3. Pass pre-resolved credentials to `streamBedrock` via `options.credentials` (new field from step A).
4. Pass inference profile ARN as `options.modelIdOverride` (re-added in step A).
5. Keep `AWS_CORP_PROFILE` / `AWS_CORP_SSO_SESSION` / `AWS_CORP_REGION` env var reads.
6. Keep `aws-corp` entry in `serviceProviderMap` in `stream.ts`.

#### C. Verify `@aws-sdk/credential-provider-sso` stays in `packages/ai/package.json`

Upstream removed the Bedrock API client packages but `aws-corp.ts` only needs the credential provider. After rebase, check:
```bash
grep "credential-provider-sso\|shared-ini-file-loader" packages/ai/package.json
```
If either is missing (upstream may have pruned the dep list), re-add them.

## Auth Broker: Does It Replace aws-corp Auth?

**Short answer: No — it solves an orthogonal problem.**

### What auth broker does

Auth broker is a **centralized OAuth credential vault** for multi-client, multi-machine deployments:

- `omp auth-broker serve` runs on a central host, holds OAuth refresh tokens in SQLite, performs background token refreshes
- `omp auth-gateway serve` is a forward proxy (same host) — accepts OpenAI/Anthropic wire formats, injects the broker-resolved access token, forwards to the real provider
- Clients (containerized omp, CI, IDE plugins, robomp) connect via `OMP_AUTH_BROKER_URL` + bearer token — they never see raw OAuth credentials
- Designed for: shared team deployments, CI pipelines, Docker containers, air-gapped omp instances

### What aws-corp does that broker cannot replace

| Capability | Auth Broker | aws-corp |
|---|---|---|
| AWS SigV4 request signing | ✗ — only handles OAuth/API-key providers | ✅ |
| AWS SSO OIDC device-auth flow | ✗ | ✅ |
| `~/.aws/credentials` profile resolution | ✗ | ✅ (via `@aws-sdk/credential-provider-sso`) |
| Inference profile ARN construction | ✗ | ✅ (`toInferenceProfileArn`) |
| Bedrock Converse Stream endpoint | Only via gateway passthrough if target is OpenAI/Anthropic wire-compatible | ✅ natively |

The broker is a credential *distribution* system for HTTP-bearer OAuth providers. Bedrock uses **SigV4** (AWS Signature Version 4), not OAuth bearer tokens. The broker has no concept of SigV4 and no path to inject AWS credentials into a Bedrock request. `aws-corp.ts` must stay.

### Could auth broker be used to centralize corp SSO token distribution?

Theoretically yes — if you ran a custom broker plugin that resolves AWS SSO tokens and exposes them as a `credential_process` output. But that is a significant infrastructure build, not something upstream supports today. The current corp setup (SSO token cached locally in `~/.aws/sso/cache/`) is simpler and sufficient.

### Corp config: auth broker stays null

```yaml
auth:
  broker:
    url: ~    # not used — aws-corp handles AWS auth directly
    token: ~
```

## Lessons from v15.0.0 Merge (apply here)

- **`--ours` = aws-corp, `--theirs` = upstream** during `git merge origin/main`. (Opposite of rebase — `--ours` is always the branch you're on.)
- **Pre-merge diff the conflict files line by line before touching them.** "Different sections" claims must be verified.
- **Python tests first** after merge — faster to catch mmemory regressions than TS cascades.
- **Delegate TS error fixing to parallel subagents** — group by file, one subagent per cluster.
- **`git checkout HEAD -- <file>` is safe during merge** (HEAD = aws-corp tip). But prefer `git checkout aws-corp -- <file>` for explicitness on our files.
- **`m-utils.ts`** — "deleted by them, modified by us" conflict marker. Resolve by keeping ours: `git checkout aws-corp -- packages/coding-agent/src/utils/m-utils.ts`.
- **mmemory + mprune**: these files never existed on upstream. Merge will keep them automatically — no recovery needed. Verify post-merge with `ls packages/coding-agent/src/tools/mmemory/`.
